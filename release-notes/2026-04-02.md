# OpenClaw v2026.4.2 版本發佈說明

[GitHub Release](https://github.com/openclaw/openclaw/releases/tag/v2026.4.2)

## ⚠️ 升級前必讀

### Breaking Changes 摘要

| 項目 | 影響 | 行動 |
|------|------|------|
| Plugins/xAI：`x_search` 設定路徑遷移 | `tools.web.x_search.*` 設定失效；xAI Web Search 停止運作 | 執行 `openclaw doctor --fix` 自動遷移至 `plugins.entries.xai.config.xSearch.*` |
| Plugins/web fetch：Firecrawl `web_fetch` 設定路徑遷移 | `tools.web.fetch.firecrawl.*` 設定失效；Firecrawl web fetch 停止運作 | 執行 `openclaw doctor --fix` 自動遷移至 `plugins.entries.firecrawl.config.webFetch.*` |

### 新功能亮點

- **Task Flow 基礎架構恢復** — 支援 managed/mirrored 同步模式、持久化流程狀態與 `openclaw flows` 操作原語
- **Android Assistant 整合** — 可從 Google Assistant 觸發 OpenClaw 並傳入提示
- **Feishu Drive 文件協作** — 新增 Drive comment-event 流程與 `feishu_drive` 評論操作
- **Matrix `m.mentions` 規範支援** — 跨訊息類型正確發送提及通知
- **Exec 預設 YOLO 模式** — gateway/node host exec 預設 `security=full, ask=off`，無需手動確認

## 功能更新 (Features) - by Star Rating

### ⭐⭐⭐ Plugins/xAI：`x_search` 設定路徑遷移 ([#59674](https://github.com/openclaw/openclaw/pull/59674))
- **用途**: 將 xAI `x_search` 設定從 legacy core `tools.web.x_search.*` 遷移至 plugin-owned `plugins.entries.xai.config.xSearch.*`，並統一 auth 至 `plugins.entries.xai.config.webSearch.apiKey` / `XAI_API_KEY`
- **解決問題**: 設定散落於 core 與 plugin 兩處，升級後路徑不一致導致 xAI Web Search 失效
- **影響**: Breaking Change；執行 `openclaw doctor --fix` 可自動遷移，無需手動修改設定

```text
┌─────────────────────────────┐
│  舊路徑                      │
│  tools.web.x_search.*       │
└─────────────────────────────┘
             │
             ▼  openclaw doctor --fix
┌─────────────────────────────────────────┐
│  新路徑                                  │
│  plugins.entries.xai.config.xSearch.*   │
│  auth: webSearch.apiKey / XAI_API_KEY   │
└─────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────┐
│  xAI Web Search 正常運作     │
└─────────────────────────────┘
```

### ⭐⭐⭐ Plugins/web fetch：Firecrawl `web_fetch` 設定路徑遷移 ([#59465](https://github.com/openclaw/openclaw/pull/59465))
- **用途**: 將 Firecrawl `web_fetch` 設定從 `tools.web.fetch.firecrawl.*` 遷移至 `plugins.entries.firecrawl.config.webFetch.*`，並透過新 fetch-provider boundary 路由 fallback
- **解決問題**: Firecrawl 設定耦合於 core，升級後 web fetch fallback 路徑失效
- **影響**: Breaking Change；執行 `openclaw doctor --fix` 自動遷移

```text
┌──────────────────────────────────┐
│  舊路徑                           │
│  tools.web.fetch.firecrawl.*     │
└──────────────────────────────────┘
             │
             ▼  openclaw doctor --fix
┌──────────────────────────────────────────────┐
│  新路徑                                       │
│  plugins.entries.firecrawl.config.webFetch.* │
└──────────────────────────────────────────────┘
             │
             ▼
┌──────────────────────────────────┐
│  fetch-provider boundary         │
│  統一路由 web_fetch fallback      │
└──────────────────────────────────┘
```

### ⭐⭐⭐ Tasks/Task Flow：基礎架構恢復 ([#58930](https://github.com/openclaw/openclaw/pull/58930))
- **用途**: 恢復 core Task Flow substrate，支援 managed/mirrored 同步模式、durable flow state/revision tracking，以及 `openclaw flows` inspection/recovery 操作原語
- **解決問題**: 背景 orchestration 無法持久化，也無法獨立於 plugin authoring layer 進行操作與恢復
- **影響**: 外部 orchestrator 可持久化並操作 background Task Flow，提升系統可靠性

```text
┌──────────────────┐     ┌──────────────────┐
│  Plugin / Agent  │────►│  Task Flow API   │
└──────────────────┘     └──────────────────┘
                                  │
              ┌───────────────────┼───────────────────┐
              ▼                   ▼                   ▼
   ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
   │  managed mode    │  │  mirrored mode   │  │  openclaw flows  │
   │  (durable state) │  │  (sync)          │  │  inspect/recover │
   └──────────────────┘  └──────────────────┘  └──────────────────┘
```

### ⭐⭐ Tasks/Task Flow：managed child task spawning ([#59610](https://github.com/openclaw/openclaw/pull/59610))
- **用途**: 新增 managed child task spawning 與 sticky cancel intent，外部 orchestrator 可立即停止排程，等待 active child tasks 完成後 parent Task Flow 自動進入 `cancelled` 狀態
- **解決問題**: 取消操作無法即時傳播，child tasks 繼續執行導致狀態不一致
- **影響**: orchestrator 可精確控制 Task Flow 生命週期

### ⭐⭐ Plugins/Task Flow：`api.runtime.taskFlow` seam ([#59622](https://github.com/openclaw/openclaw/pull/59622))
- **用途**: 新增 bound `api.runtime.taskFlow` seam，plugin 與 trusted authoring layer 可從 host-resolved OpenClaw context 建立並驅動 managed Task Flow，無需每次傳入 owner identifier
- **解決問題**: plugin 無法直接操作 Task Flow，需繞過 host context
- **影響**: plugin 開發更簡潔，Task Flow 整合更安全

### ⭐⭐ Android/assistant：entrypoints 與 App Actions ([#59596](https://github.com/openclaw/openclaw/pull/59596))
- **用途**: 新增 assistant-role entrypoints 與 Google Assistant App Actions metadata，Android 可從 assistant 觸發 OpenClaw 並將提示傳入 chat composer
- **解決問題**: Android 用戶無法透過 Google Assistant 快速啟動 OpenClaw
- **影響**: 提升 Android 平台整合深度，支援語音觸發工作流程

### ⭐⭐ Exec 預設 YOLO 模式
- **用途**: gateway/node host exec 預設請求 `security=full, ask=off`，並對齊 host approval-file fallbacks 與 docs/doctor reporting
- **解決問題**: 預設需手動確認每個 exec 操作，影響自動化工作流程效率
- **影響**: 自動化場景下無需額外設定即可無提示執行；需注意安全邊界

### ⭐⭐ Providers/runtime：replay hook surfaces ([#59143](https://github.com/openclaw/openclaw/pull/59143))
- **用途**: 新增 provider-owned replay hook surfaces，涵蓋 transcript policy、replay cleanup、reasoning-mode dispatch
- **解決問題**: provider 無法自訂 replay 行為，導致 transcript 處理不一致
- **影響**: provider 可精確控制 replay 流程，提升多 provider 場景下的一致性

### ⭐⭐ Plugins/hooks：`before_agent_reply` ([#20067](https://github.com/openclaw/openclaw/pull/20067))
- **用途**: 新增 `before_agent_reply` hook，plugin 可在 LLM 回覆前以 synthetic reply 短路整個回覆流程
- **解決問題**: plugin 無法在 LLM 生成回覆前介入，限制了 inline action 後的自訂回應能力
- **影響**: plugin 可實現更豐富的 pre-reply 邏輯，如快取回覆、條件攔截等

### ⭐⭐ Feishu/Drive comment-event flow ([#58497](https://github.com/openclaw/openclaw/pull/58497))
- **用途**: 新增專屬 Drive comment-event flow，支援 comment-thread context resolution、in-thread replies，以及 `feishu_drive` comment actions
- **解決問題**: Feishu Drive 文件評論無法觸發 agent 回覆，缺乏文件協作整合
- **影響**: 支援 Feishu Drive 文件協作工作流程，agent 可直接回覆文件評論

### ⭐⭐ Matrix：`m.mentions` 規範支援 ([#59323](https://github.com/openclaw/openclaw/pull/59323))
- **用途**: 跨文字發送、媒體 caption、編輯、poll fallback text、action-driven edits 發送符合規範的 `m.mentions` metadata
- **解決問題**: Matrix 提及通知在 Element 等客戶端無法可靠觸發
- **影響**: Matrix 用戶可正確收到提及通知

### ⭐⭐ Diffs：plugin-owned `viewerBaseUrl` ([#59341](https://github.com/openclaw/openclaw/pull/59341))
- **用途**: 新增 plugin-owned `viewerBaseUrl`，viewer 連結可使用穩定 proxy/public origin，無需每次 tool call 傳入 `baseUrl`
- **解決問題**: viewer 連結依賴動態 baseUrl，在 proxy 環境下不穩定
- **影響**: diff viewer 連結更穩定，適合 proxy/CDN 部署場景

### ⭐ Channels/session routing
- **用途**: provider-specific session conversation grammar 移至 plugin-owned session-key surfaces，保留 Telegram topic routing 與 Feishu scoped inheritance
- **解決問題**: session routing 邏輯耦合於 core，難以擴展
- **影響**: 架構更模組化，provider 可自訂 session routing 行為

### ⭐ Agents/compaction：model override 一致性 ([#56710](https://github.com/openclaw/openclaw/pull/56710))
- **用途**: `agents.defaults.compaction.model` 跨 manual `/compact` 與所有 context-engine compaction 路徑一致解析
- **解決問題**: compaction model override 僅在部分路徑生效
- **影響**: compaction 行為更可預測

### ⭐ Agents/compaction：`notifyUser` opt-in ([#54251](https://github.com/openclaw/openclaw/pull/54251))
- **用途**: 新增 `agents.defaults.compaction.notifyUser`，`🧹 Compacting context...` 通知改為 opt-in
- **解決問題**: compaction 通知強制顯示，干擾部分使用場景
- **影響**: 可依需求關閉 compaction 通知

### ⭐ WhatsApp/reactions：`reactionLevel` guidance
- **用途**: 新增 `reactionLevel` guidance for agent reactions
- **解決問題**: agent reaction 行為缺乏明確指引
- **影響**: WhatsApp reaction 行為更一致

### ⭐ Exec approvals/channels：DM-first native chat approvals
- **用途**: 支援頻道自動推斷 approver，啟用 DM-first native chat approvals，同時保持 channel fanout 明確
- **解決問題**: approval 設定需手動指定，無法自動從現有 owner config 推斷
- **影響**: 簡化 approval 設定流程

## 安全性提升 (Security) - by Star Rating

### ⭐⭐⭐ Providers/transport policy 集中化 ([#59682](https://github.com/openclaw/openclaw/pull/59682))
- **用途**: 集中化 request auth、proxy、TLS、header shaping 跨 shared HTTP、stream、websocket paths；封鎖不安全 TLS/runtime transport override；保持 proxy-hop TLS 與 target mTLS 分離
- **解決問題**: transport 設定分散，不安全的 TLS override 可能被利用繞過安全邊界
- **影響**: 所有 provider HTTP 路徑的 transport 安全性統一強化，降低設定錯誤風險

```text
┌──────────────────────────────┐
│  Provider HTTP Request       │
└──────────────────────────────┘
             │
             ▼
┌──────────────────────────────┐
│  Centralized Transport Policy│
│  • auth shaping              │
│  • proxy config              │
```
